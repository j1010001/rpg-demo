# Contract: GameState Schema

**Module**: `js/game.js` (owner) | consumed by all other modules
**Branch**: `001-retro-dungeon-crawler`

---

## Overview

`window.GameState` is the single shared mutable state object. All modules read from and write to this object directly. No copies are made; modules operate on the live reference.

---

## Schema

```js
window.GameState = {
  // Current floor's dungeon
  dungeon: {
    tiles: Tile[][],        // [y][x] indexed; dimensions = config.dungeonWidth × config.dungeonHeight
    rooms: Room[],          // All rooms generated by BSP this floor
    width: number,          // = config.dungeonWidth
    height: number,         // = config.dungeonHeight
    stairPos: { x: number, y: number }  // Position of the STAIR_DOWN tile
  },

  // Player — persists across floors
  player: {
    x: number,
    y: number,
    hp: number,
    maxHp: number,
    attack: number,
    defense: number,
    level: number,          // Display only; = floor (FR-016)
    floor: number,          // 1–10
    inventory: Item[],      // Max config.inventoryCap items
    equippedWeapon: Item | null,
    equippedArmor: Item | null,
    enemiesDefeated: number
  },

  enemies: Enemy[],         // Alive enemies on current floor only
  items: Item[],            // Items on floor (not in inventory); item.x/item.y are set

  phase: "PLAYING" | "GAME_OVER" | "VICTORY",

  log: string[],            // HUD combat log; newest entry last; max 8 entries retained

  config: {
    dungeonWidth: 80,
    dungeonHeight: 50,
    sightRadius: 5,         // Default for D3-deferred; change this constant to tune FOV
    maxFloors: 10,
    inventoryCap: 10,
    minPartition: 8,        // BSP: minimum partition size before stopping recursion
    minRoomSize: 4          // BSP: minimum room dimension within a partition
  }
}
```

---

## Tile Schema

```js
{
  type: "WALL" | "FLOOR" | "DOOR" | "STAIR_DOWN" | "ITEM_GROUND" | "EMPTY",
  visibility: "HIDDEN" | "SEEN" | "LIT",
  glyph: string,            // Single character; set from type, overridden by entity presence
  entity: Enemy | Item | null
}
```

---

## Enemy Schema

```js
{
  id: number,
  x: number,
  y: number,
  type: "GOBLIN" | "ORC" | "TROLL",
  glyph: "g" | "o" | "T",
  hp: number,
  maxHp: number,
  attack: number,
  sightRange: number,
  alive: boolean
}
```

---

## Item Schema

```js
{
  id: number,
  type: "POTION" | "WEAPON" | "ARMOR",
  glyph: "!" | "/" | "[",
  name: string,
  effect: number,           // HP restored / attack bonus / defense bonus
  x: number | null,         // null when in inventory
  y: number | null          // null when in inventory
}
```

---

## Mutation Rules

- Only `game.js` initializes `GameState` and transitions `phase`.
- Only `dungeon.js` writes `GameState.dungeon`.
- Only `fov.js` writes `tile.visibility` fields.
- Only `player.js` writes `GameState.player` fields.
- Only `enemy.js` writes `enemy.alive`, `enemy.hp`, `enemy.x`, `enemy.y`.
- Only `combat.js` writes `player.hp` (damage from enemies) and calls the log append.
- Only `ui.js` reads `GameState.log` for display; `combat.js` pushes to `GameState.log`.
- `renderer.js` is read-only with respect to GameState — it never mutates state.

These are conventions, not enforced by the runtime. Violating them introduces hard-to-debug state bugs.
